🧠 The DOM & JavaScript as a Weapon

The browser does not just display HTML — it builds a live, programmable tree from it. JavaScript can read, rewrite, and weaponise every node in that tree. Understanding this is the key to understanding every XSS attack.

⏱️ ~20 minutes 📖 Reading + Interactive ⚡ Live demos
🌳 The DOM: The Browser's Live Model of Your Page

When a browser receives an HTML file, it does not just paint it to the screen and forget it. It parses the entire document and builds it into a tree of objects in memory called the Document Object Model (DOM). Every tag becomes a node. Every attribute becomes a property. Every piece of text becomes a text node.

This tree is live — JavaScript can read it, modify it, add to it, and delete from it at any time. That is how modern websites work: buttons update content without reloading, forms validate in real time, menus appear and disappear. The DOM is the API between HTML and JavaScript.

From a security perspective, the DOM is the battlefield. Every XSS attack is ultimately an attack on the DOM — injecting instructions that the browser will execute with the full trust and permissions of the page.

ℹ️ Source vs Sink — the two halves of DOM XSS
A source is where attacker-controlled data enters JavaScript — e.g. location.hash, a form input value, document.referrer. A sink is where that data gets written into the DOM — e.g. innerHTML, document.write(). DOM XSS exists when data flows from a source to a dangerous sink without sanitization.
⚔️ innerHTML vs textContent — The Most Important Difference in Web Security

There is one decision developers make constantly — and one of the two options creates XSS vulnerabilities every time. The choice between innerHTML and textContent looks trivial in code. The difference in behaviour is enormous.

🔐 The Rule: Never pass user input to innerHTML
If you must insert HTML dynamically, sanitize first with DOMPurify: div.innerHTML = DOMPurify.sanitize(userInput). But in most cases textContent is what you actually need — and it is always safe. Try putting <img src=x onerror=alert('XSS')> in the innerHTML panel above to see the difference.
💥 DOM-Based XSS — The Attack That Never Touches the Server

In Module 1 you saw Reflected XSS (payload in URL, reflected by server) and Stored XSS (payload saved to database). DOM-based XSS is different — the server is completely uninvolved. The JavaScript running in the page reads attacker-controlled data and writes it into the DOM using a dangerous sink.

This means server-side sanitization cannot stop it. The server sends a perfectly clean HTML page. The vulnerability lives entirely in the client-side JavaScript.

1

Attacker crafts a URL with payload in the fragment

https://shop.com/search#<img src=x onerror=alert(document.cookie)>
The #fragment is never sent to the server — the browser keeps it locally.

2

Victim clicks the link — page loads normally

Server receives GET /search — no payload visible. Returns clean HTML. No filter triggers.

3

Client-side JavaScript reads the fragment — the source

const query = location.hash.slice(1);

4

JavaScript writes it into the DOM via innerHTML — the sink

document.getElementById('results').innerHTML = query;
Browser parses the injected HTML. The onerror fires. Cookie stolen.

5

The fix — one word change

document.getElementById('results').textContent = query;
Payload becomes harmless text. Zero XSS.

StepWhat they doWhat they look for
1 — Map sources Find where attacker data enters JS location.hash, location.search, document.referrer, window.name, postMessage
2 — Map sinks Find dangerous write functions innerHTML, outerHTML, document.write, eval, setTimeout
3 — Trace the flow Check if data travels source → sink without sanitization Any assignment or concatenation between source and sink
4 — Craft a payload Build a URL placing JS payload at the source #<img src=x onerror=alert(1)> — the standard first test
