🧪 Lab Overview
This lab simulates a real-world vulnerable web application — a simple guestbook where users can leave messages. Every guest entry is "stored" and displayed to all visitors. You will use everything from Module 1: HTML structure, forms, GET/POST, and input handling.

The application runs entirely in your browser using localStorage to simulate a database. No server is involved — but the vulnerability and the fix are identical to what you would find on a real server.

3
Inject HTML into the Guestbook
Not started

Now that you know the guestbook displays your message directly in HTML, test whether it sanitizes your input. Submit a message that contains an HTML tag.

Try each of these in the message field of the guestbook above (make sure it is in Vulnerable mode):

PayloadExpected result if vulnerable
<b>I am bold</b>Message appears in bold text
<h1>BIG TEXT</h1>Message appears as a large heading
<marquee>scrolling</marquee>Text scrolls across the entry (old but fun)
<img src=x onerror="this.style.display='none'">hidden imageImage tag silently fails — confirms tag injection
⚠️ What this tells you
If your HTML renders as formatted text instead of appearing as literal characters like &lt;b&gt;, the application is not encoding output. This means the browser is interpreting your input as real HTML — HTML injection is confirmed. The next step is JavaScript.
4
Achieve Stored XSS
Not started

HTML injection is confirmed. Now escalate to Stored XSS — inject JavaScript that executes every time a visitor loads the guestbook page.

In the guestbook above (Vulnerable mode), submit a message containing one of these payloads:

🧪 XSS Payloads — paste into the Message field
<script>alert('XSS by ' + document.title)</script>
<img src=x onerror="alert('Stored XSS!')">
<svg onload="alert('XSS via SVG')"></svg>
🔐 Why this is Stored XSS — not Reflected
In Reflected XSS, the payload is in the URL and only affects the person who clicks the link. In Stored XSS, the payload is saved to the database (here: localStorage) and executes for every visitor who loads the page — including administrators, making it far more dangerous. Cookie theft via Stored XSS can compromise every active session at once.
5
Fix the Vulnerability
Not started

Now switch the guestbook to Safe Mode using the toggle in the guestbook panel above. Submit the same XSS payloads you used in Task 4 and observe what happens.

💡 What Safe Mode does
Safe Mode runs every message through an HTML encoding function before inserting it into the page. The characters < and > are converted to &lt; and &gt; — harmless text the browser displays literally, not as HTML tags. The payload is stored the same way but rendered differently.

Here is the difference between the two approaches in code:

output-encoding-comparison.js
ℹ️ This is called Output Encoding
The fix is applied at the point where data is output to the HTML — not at the input stage. This is a fundamental principle: encode data for the context where it will be used. HTML context → HTML encoding. JavaScript context → JavaScript encoding. SQL context → parameterised queries.
