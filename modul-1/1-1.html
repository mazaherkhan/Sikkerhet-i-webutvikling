Module 1 · Sub-module 1.1

🌐 From URL to Webpage

Every web attack starts with a request. Before you can break something, you need to understand exactly what happens the moment a user types a URL and hits Enter.

⏱️ ~20 minutes 📖 Reading + Interactive 🧠 4-question quiz

🎯 What you will learn

  • What a URL is and what each part of it means
  • How HTTP requests and responses work
  • The step-by-step journey from URL to rendered page
  • Where in this process an attacker can interfere
🔗 What is a URL?

A URL (Uniform Resource Locator) is the address of a resource on the internet. You use one every time you visit a website, load an image, or submit a form. But a URL is not just a web address — it is a structured instruction that tells the browser exactly what to request, from where, and how.

Every URL has distinct parts, and as a security professional you need to read them precisely — because attackers hide malicious payloads inside URLs, especially in the query string.

https://www.example.com/search?q=hello&page=2#results

← Click any coloured part of the URL above to see what it means.

🔐 Security Note
The query string (the part after ?) is the most common injection point for Reflected XSS. An attacker crafts a URL with a malicious payload in the query string, then tricks a victim into clicking it. The server reflects the payload back in the HTML response — and the browser executes it. You will practice this in Module 3.
📡 HTTP — The Language of the Web

When your browser has parsed the URL, it sends an HTTP request to the server. HTTP (HyperText Transfer Protocol) is the set of rules that governs how browsers and servers communicate. Think of it as a conversation: the browser asks a question (request), and the server gives an answer (response).

This is a real HTTP request. Your browser sends something like this every time you visit a page.

Browser → Server 
GET /search?q=hello HTTP/1.1
Host:            www.example.com
User-Agent:      Mozilla/5.0 (Windows NT 10.0)
Accept:          text/html
Cookie:          sessionid=abc123xyz  ← sent automatically with every request
Referer:         https://www.example.com/home
🔐 Security Note
Notice the Cookie header — the browser sends your session cookie automatically with every request. This is why XSS is so dangerous: if an attacker can run JavaScript in your browser, they can steal this cookie and hijack your session.

The server sends back a response containing a status code, headers, and the HTML body.

Server → Browser 
HTTP/1.1 200 OK
Content-Type:   text/html; charset=UTF-8
Set-Cookie:     sessionid=abc123xyz; HttpOnly
X-Frame-Options: DENY

<!DOCTYPE html>
<html>
  <body>
    <h1>Results for: hello</h1>  ← user input reflected here
  </body>
</html>
⚠️ Notice
The search term hello appears directly inside the HTML response. If instead of hello the user sent <script>alert(1)</script> — and the server did not sanitize it — the browser would execute that script. That is Reflected XSS in one sentence.

The two most important HTTP methods for web security are GET and POST.

🔐 Security Note
GET parameters in the URL can be crafted by an attacker and sent as a link to a victim. This is exactly how Reflected XSS attacks are delivered in the real world.
🔄 The Full Journey — Step by Step

Let's trace exactly what happens from the moment a user types https://www.example.com/search?q=hello and hits Enter. Each step is an opportunity — for both the developer and the attacker.

1

DNS Lookup

The browser asks a DNS server: "What is the IP address of www.example.com?" It receives back an IP like 93.184.216.34. This is like looking up a phone number in a directory.

2

TCP Connection + TLS Handshake (HTTPS)

The browser establishes a secure encrypted connection to the server. This is what the https:// prefix means — the traffic is encrypted in transit. http:// (without S) is unencrypted and can be intercepted.

3

HTTP Request Sent

The browser sends the GET request to the server, including the query string ?q=hello, cookies, and headers.

4

Server Processes the Request

The server reads the query string, looks up results, and builds an HTML response. This is the critical moment — if the server inserts the user's input directly into the HTML without sanitizing it, a vulnerability is introduced.

5

HTTP Response Received

The browser receives the HTML document, CSS files, JavaScript files, and images. The status code tells the browser if everything went well (200 OK), if the page moved (301 Redirect), or if something went wrong (404 Not Found, 500 Server Error).

6

Browser Parses HTML → Builds the DOM

The browser reads the HTML top to bottom and constructs a tree structure in memory called the DOM (Document Object Model). You will learn exactly what the DOM is in Module 2.

7

JavaScript Executes

Any <script> tags are executed at this point. If the HTML contains malicious JavaScript injected by an attacker, it runs here — with full access to cookies, the DOM, and the user's session.

8

Page is Rendered

The browser paints the final page on screen. The user sees the result — unaware that in step 7, malicious code may have already executed and stolen their session cookie.

💡 The Attacker's Perspective
An attacker does not need to break the encryption in step 2 or intercept the network in step 3. They only need to inject a malicious payload in step 3 (via a crafted URL or form input), have it survive step 4 unsanitized, and it will automatically execute in step 7 — in the victim's browser.
📊 HTTP Status Codes You Must Know

Status codes are the server's way of telling the browser what happened. As a security tester, you read these constantly.
