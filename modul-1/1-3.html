📝 Forms & User Input

Forms are how users talk to web applications. They are also the primary way attackers send malicious data. Understanding exactly how forms work — and where data goes — is essential before you can test for XSS.

⏱️ ~25 minutes 📖 Reading + Interactive ⚡ Live code editor 🧠 4-question quiz

  • How HTML forms work and how data is sent to a server
  • The difference between GET and POST — and why it matters for attacks
  • Every important input type and its security implications
  • How "hidden" fields and client-side controls can be bypassed
  • How to trace user input from browser to server to response
📋 What is an HTML Form?

An HTML form is a section of a webpage that collects information from the user and sends it to a server. Every time you log in, search for something, post a comment, or fill out a registration — you are using an HTML form.

From an attacker's perspective, every form is an input channel — a place where attacker-controlled data enters the application. Identifying and probing forms is one of the first steps in any web application penetration test.

ℹ️ The attacker's first question
When a penetration tester opens a web application, the first question is: "Where can I send data in?" Forms are the most visible answer. But as you will learn later, data also enters via URL parameters, cookies, HTTP headers, and JavaScript — all of which are equally valid injection points.

⚙️ GET vs POST — An Attacker's Perspective

The HTTP method determines where the form data travels when submitted. This is one of the most important distinctions in web security.

When method is GET, the form data is appended to the URL as a query string:

Browser → Server (GET)
GET /search?q=boots&category=products HTTP/1.1
Host: shop.example.com
Cookie: sessionid=abc123 ← sent automatically
🔐 Why GET is dangerous for XSS delivery
The query string ?q=boots is part of the URL. An attacker can replace boots with <script>alert(1)</script>, craft a link, and email it to a victim. One click — the payload is in the URL, the server reflects it back in the HTML, and it executes. This is a Reflected XSS attack delivered via GET parameter.

When method is POST, the form data is in the request body — not the URL:

Browser → Server (POST)
POST /search HTTP/1.1
Host: shop.example.com
Content-Type: application/x-www-form-urlencoded
Cookie: sessionid=abc123

q=boots&category=products ← body, not URL
⚠️ POST is harder to share — but not safe
POST data is not in the URL, so an attacker cannot simply send a link. However, POST requests can still carry XSS payloads — an attacker uses a crafted HTML page that auto-submits a POST form when the victim visits it. POST also does not protect against Stored XSS: if the POST data is saved to a database and later displayed to users, the attack still works.

In Burp Suite (which you will use in Module 4), you intercept the request between browser and server and modify it before it is sent:

Burp Suite Intercepted Request — before sending to server
POST /search HTTP/1.1
Host: shop.example.com
Content-Type: application/x-www-form-urlencoded
Cookie: sessionid=abc123

# You change "boots" to your XSS payload:
q=<img src=x onerror=alert(document.cookie)>&category=products
💡 Why Burp Suite is essential
With Burp, you can intercept any request — GET or POST — and modify every parameter, header, and cookie before it reaches the server. This is how professional penetration testers test for XSS and other injection vulnerabilities. You will use Burp in Module 4.

🔐 The most important rule in web security

      Client-side controls are not security controls.
      Every validation, every restriction, every dropdown limit you see in a browser
      can be bypassed by modifying the request directly — using Burp Suite, curl, or even
      browser DevTools. The only real security happens on the server side.
    



    
    
👻 Hidden Fields — The Illusion of Security


    

      Hidden input fields (<input type="hidden">) are widely used by developers
      to pass data between pages without showing it to the user. A common (and dangerous) pattern
      is to use them to pass security-relevant values like user roles, prices, or session tokens.

🔐 Real-world impact of hidden field manipulation

      In real penetration tests, hidden field manipulation has been used to:
      
• Change product prices to £0.01 before checkout
      
• Escalate user roles from role=user to role=admin
      
• Bypass discount limits by modifying discount codes
      
• Inject XSS payloads into fields the developer never thought would be tampered with
    



    
    
🔄 Tracing Data: From Browser to Server to Response


    

      The most important skill for finding XSS vulnerabilities is being able to
      trace where user input travels through the application.
      The questions to always ask are: "Where did I send this data?" and "Where does it come back?"
    


    

      


        

          
① User types input into a form field

          
Example: user types <script>alert(1)</script> into a search box

        

        
↓


        

          
② Browser sends HTTP request

          
GET: /search?q=<script>alert(1)</script>

             POST: payload in request body — intercept with Burp

        

        
↓


        

          
③ Server receives and processes the input

          
Server reads q parameter, searches the database, prepares an HTML response

        

        
↓


        

          
④ ⚠️ Server inserts input into HTML without sanitizing

          
Response HTML: <p>Results for: <script>alert(1)</script></p>

        

        
↓


        

          
⑤ 🚨 Browser renders response — script executes

          
The browser sees a <script> tag, executes it. Full XSS achieved.
          Attacker can now steal cookies, hijack session, or deface the page.

        

        
↓


        

          
✅ What should happen: Server sanitizes the input

          
Response HTML: <p>Results for: &lt;script&gt;alert(1)&lt;/script&gt;</p>

          Browser renders it as harmless text — the script never executes.

        


      

    


    

      
💡 Your testing checklist for any form field

      1. What HTTP method does the form use? (GET or POST)

      2. What is the action URL? (Where does data go?)

      3. Are there any hidden fields? (What are their names and values?)

      4. Does the server reflect the input anywhere in the response?

      5. Is the reflected input inside a tag, inside an attribute, or inside JavaScript?
